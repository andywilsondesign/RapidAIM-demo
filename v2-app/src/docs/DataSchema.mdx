# RapidAIM UI Data Schema

This document defines the schema representing dynamic data versus static layout text for the RapidAIM front-end components.

## Core Entities

### 1. Ranch (Organization Level)
Represents an agricultural property managed by an organization.

<Source
  language="typescript"
  dark
  code={`interface Ranch {
  id: string; // e.g. "R1"
  name: string; // e.g. "Sunshine Valley Ranch"
  urgencyLevel: 'STABLE' | 'MONITOR' | 'WARNING' | 'CRITICAL';
  currentCount: number;
  totalBlocks: number;
  activeSensors: number;
  uptimePercentage: number;
  trendPercentage: number; // e.g. 24 (positive implies increase)
  description: string; // AI generated insight or manual note
}`}
/>

### 2. Block (Field Level)
Represents a specific block within a Ranch.

<Source
  language="typescript"
  dark
  code={`interface Block {

  name: string; // e.g. "Block 4"

  currentPestCount: number;

## Component Mapping (Dynamic vs Static)

When building components, standard labels remain **static** (hardcoded in the UI or localized), whereas the data driving them is **dynamic** (passed via props).

### StatCard Molecule
- **Static:** The label (e.g., "Current Count", "Uptime").
- **Dynamic:** The title (e.g., "Female Navel Orangeworm"), value (e.g., \`98\`), and trend (e.g., \`+24%\`).

### Alerts (Global & Inline)
- **Static:** Form field structure, error icon.
- **Dynamic:** The specific error message (e.g., "Email is required", "Service currently degraded in region X"), Alert severity (\`info\`, \`warning\`, \`error\`, \`success\`).

### Account Settings Form
- **Static:** Input labels ("Organization Name", "Email Address").
- **Dynamic:** User inputs (bound to state), validation state (true/false).
